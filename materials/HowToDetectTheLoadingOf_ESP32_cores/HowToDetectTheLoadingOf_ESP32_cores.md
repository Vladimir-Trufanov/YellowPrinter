## Как определить загрузку ядер ESP32

### [Тест загрузки процессора ESP32 без специальной настройки](#)

### [Монитор использования процессора ESP32](#)

Определить загрузку ядер ESP32 можно несколькими способами — выбор зависит от того, что именно вам нужно: понять, на каком ядре сейчас работает код, или получить процент загрузки для мониторинга. 

#### 1. Узнать, на каком ядре выполняется код

Если вам нужно понять, какое ядро (0 или 1) сейчас обрабатывает ту или иную часть программы, используйте функцию ***xPortGetCoreID()***. Она возвращает идентификатор текущего ядра. 

Пример для Arduino IDE:

```
void setup() 
{
  Serial.begin(115200);
  Serial.print("setup() running on core ");
  Serial.println(xPortGetCoreID());
}

void loop() 
{
  Serial.print("loop() running on core ");
  Serial.println(xPortGetCoreID());
  delay(1000);
}
```

После загрузки скетча откройте монитор последовательного порта — вы увидите, на каком ядре выполняются функции setup() и loop(). 

Важное уточнение: ***в Arduino IDE по умолчанию код запускается на ядре 1***. 

При этом ***ядро 0 часто задействовано для радиочастотных (RF) операций***, поэтому для некоторых задач (например, обработки сетевых запросов) может быть выгоднее явно назначить задачу ядру 0. Чтобы закрепить задачу за конкретным ядром, используйте xTaskCreatePinnedToCore(). 

В среде ESP-IDF ядра иногда обозначают как PRO_CPU (ядро 0) и APP_CPU (ядро 1). Логика та же: xPortGetCoreID() покажет, какое из них сейчас активно. 

#### 2. Измерить процент загрузки ядра

Если цель — не узнать, какое ядро сейчас работает, а оценить, насколько интенсивно оно используется (в процентах), подойдут другие подходы.

Компонент ***esp32_perfmon***. Это сторонний монитор, который работает так: он подключает хуки в состояние простоя и измеряет, как часто они срабатывают. Процент загрузки рассчитывается по формуле: (ожидаемые тики в простое / полученные тики в простое) * 100%. Важно: этот метод даёт лишь приблизительную оценку, его не стоит использовать в критически важной логике, где нужна абсолютная точность. Также есть нюанс: если одно ядро загружено на 100%, второе может показать отрицательное значение. 

Performance Monitor в ESP-IDF. Этот встроенный механизм позволяет профилировать функции и приложения, используя внутренние счётчики производительности ESP32. В примерах ESP-IDF (например, в каталоге ***examples/system/perfmon***) показано, как инициализировать монитор и выводить статистику. Это более точный инструмент для глубокого анализа, чем esp32_perfmon. 

#### 3. Функции FreeRTOS для анализа нагрузки и состояния задач

***Для оценки загрузки и «здоровья» задач:***

- ***uxTaskGetStackHighWaterMark(TaskHandle_t xTask)*** — возвращает «высокую
отметку» (high water mark) использования стека: сколько свободного места
осталось в худшем случае. Это не про загрузку CPU, но критично: если запас
маленький, задача может переполниться и вызвать панику.

Пример:

```
unsigned short highWater = uxTaskGetStackHighWaterMark(NULL); 
// для текущей задачи
Serial.printf("Stack free: %d words\n", highWater);
```

Полезно при отладке тяжёлых задач (WiFi, парсинг, вычисления) — как раз в
духе ваших прошлых вопросов про оптимизацию и работу с памятью.

- ***eTaskGetState(TaskHandle_t xTask)*** — возвращает текущее состояние задачи:
***Running, Ready, Blocked, Suspended, Deleted***. Можно периодически опрашивать
состояния всех задач и выводить отчёт. Это помогает понять, не «зависла» ли
какая-то задача в блокировке (например, ждёт семафор/очередь, которой никто
не даёт).

***Для замеров времени и профилирования(альтернатива vTaskGetRunTimeStats):***

- ***xTaskGetTickCount() / xTaskGetTickCountFromISR()*** — дают количество тиков с момента старта. С их помощью можно делать собственные замеры длительности
участков кода:

```
TickType_t start = xTaskGetTickCount();
// ... тяжёлый код ...
TickType_t duration = xTaskGetTickCount() - start;
Serial.printf("Duration: %d ms\n", duration * portTICK_PERIOD_MS);
```

Это простой и надёжный способ профилировать отдельные функции —
особенно если вы писали про printf и отладочный вывод.

- Ручной замер на основе ***esp_cpu_get_cycle_count()*** (ESPIDF, но можно и в
Arduino через соответствующий заголовок) — счётчик циклов CPU. Даёт очень
точные замеры коротких участков (микросекунды). Удобно для узких мест в
обработке данных или работе со флэшпамятью (вспомним ваши вопросы про
флэш и PROGMEM).

- Для получения списка задач и их параметров ***vTaskList(char *pcWriteBuffer)*** — заполняет буфер текстовым отчётом обо всех задачах: имя, приоритет, состояние, используемый стек. В Arduino это удобно выводить в Serial.

Важно: функция временно отключает прерывания, поэтому её не стоит вызывать
в частом цикле. Лучше по событию (кнопка, команда через UART/WiFi) или в
начале отладки.

```
static char taskList[1024]; 
vTaskList(taskList);
Serial.println(taskList);
```

- ***vTaskGetCPUUsageStats(char *pcWriteBuffer)*** (в некоторых версиях
FreeRTOS/ESPIDF) — даёт процентную загрузку по задачам. Если в вашей
сборке такой функции нет, её аналог легко собрать вручную на базе
vTaskGetRunTimeStats().

***Полезные вспомогательные функции (для анализа и отладки)***

- ***xPortGetCoreID()*** — уже упоминали, но это база для анализа распределения
нагрузки между ядрами. Если вы создаёте задачи с привязкой к ядру
(xTaskCreatePinnedToCore), полезно логировать, на каком ядре реально
выполняется критичный код.

- ***esp_task_wdt_feed()*** и понимание работы TASKWDT — не функция анализа,
но важный индикатор: если TASKWDT срабатывает, значит, какая-то задача
не успевает «покормиться» и, скорее всего, сильно нагружает ядро либо зависла.

Это косвенный признак перегрузки.

- Критические секции ***taskENTER_CRITICAL() / taskEXIT_CRITICAL()*** — если
нужно замерить короткий участок без вытеснения другими задачами. Полезно
при анализе гонок данных и блокировок (актуально, если вы работаете с
очередями/семафорами).

***Практические советы с учётом ваших прошлых задач***

- a) Если вы оптимизируете код под ESP32 (как в вопросах про PROGMEM, флэш,
strcatподобные функции), начните с замеров длительности критических
участков через xTaskGetTickCount или esp_cpu_get_cycle_count. Это покажет, где
именно уходит время.

- б) Для поиска «узких мест» в многозадачности используйте пару vTaskList +
eTaskGetState — так вы увидите, какие задачи чаще всего в Blocked и сколько у
них запаса стека.

- в) Чтобы понять, равномерно ли распределена нагрузка между ядрами,
комбинируйте xPortGetCoreID с логированием длительности участков на каждом
ядре.

> Если скажете, какой стек вы используете (Arduino IDE с arduino-esp32 или
> ESPIDF) и что именно хотите измерить (мгновенная загрузка, пики, средняя
> нагрузка, поиск зависаний) подскажу готовый пример кода


#### 4. Функции FreeRTOS для сбора статистики задач

FreeRTOS (реальная ОС, используемая в ESP32) предоставляет встроенные функции для анализа загрузки. Важно: перед использованием нужно включить соответствующие опции в конфигурации.

- ***vTaskGetRunTimeStats(char *pBuffer)*** — получает статистику времени выполнения всех задач в виде текстовой таблицы. Показывает, сколько времени каждая задача использовала CPU (в тиках и процентах).

```
void TaskMonitor(void *pvParameters)
{
  static char buffer[2048];
  while (true) 
  {
    vTaskGetRunTimeStats(buffer);
    Serial.println(buffer);
    vTaskDelay(pdMS_TO_TICKS(5000)); // Каждые 5 секунд
  }
}
```

- ***vTaskList(char *pBuffer)*** — выводит список всех задач с их состоянием (Running, Ready, Blocked и т. д.), приоритетом, использованием стека и номером задачи.

```
void PrintTaskList() 
{
  static char buffer[1024];
  vTaskList(buffer);
  Serial.println("Task List:");
  Serial.println(buffer);
}
```

ESP32 имеет встроенные счётчики производительности (Performance Counters), доступные через регистры.

- ***esp_cpu_get_cycle_count()*** — возвращает количество тактов процессора с момента последнего сброса. Можно использовать для замера времени выполнения критических секций кода.

```
uint32_t start = esp_cpu_get_cycle_count();
// Ваш код здесь
uint32_t end = esp_cpu_get_cycle_count();
Serial.printf("Выполнено за %u тактов\n", end - start);
```

Регистры PMU (Performance Monitoring Unit) — позволяют отслеживать конкретные события (промахи кэша, инструкции и т. д.). Требуют прямого доступа к регистрам и понимания архитектуры Tensilica LX6.

В ESP-IDF есть специализированные функции:

```
esp_timer_get_time() — возвращает время в микросекундах с момента запуска.
                       Подходит для точного замера интервалов.

esp_get_free_heap_size() — показывает свободную память кучи. 
                           Косвенно указывает на нагрузку: резкое падение 
                           может означать утечку памяти или высокую 
                           активность задач.

esp_get_maximum_free_block_size() — размер самого большого свободного блока
                                    памяти. Помогает диагностировать 
                                    фрагментацию памяти.
```

#### 5. Сторонние библиотеки и инструменты

esp32_perfmon — библиотека для расчёта загрузки CPU на основе анализа времени простоя. Подходит для Arduino IDE.

ESP-IDF Performance Monitor — встроенный инструмент для профилирования функций. Позволяет замерять время выполнения конкретных участков кода.

Wireshark + ESP-IDF Logging — отправка логов по UDP для анализа сетевого трафика и задержек.

#### 6. Мониторинг энергопотребления

Косвенный показатель нагрузки:

- esp_pm_get_mode() — текущий режим энергопотребления (Active, Light-sleep и т. д.).

- adc1_get_raw() (с внешним шунтом) — замер тока потребления. Высокая нагрузка обычно сопровождается ростом потребления.

Практический пример: комплексный мониторинг для Arduino IDE, который объединяет несколько методов:

```
void setup() 
{
  Serial.begin(115200);
  // Запускаем на ядре 0
  xTaskCreatePinnedToCore(MonitorTask, "Monitor", 4096, NULL, 1, NULL, 0); 
}

void MonitorTask(void *pvParameters)
{
  while (true) 
  {
    Serial.printf("Core: %d\n", xPortGetCoreID());
    Serial.printf("Free heap: %u bytes\n", esp_get_free_heap_size());

    static char stats[2048];
    vTaskGetRunTimeStats(stats);
    Serial.println(stats);

    vTaskDelay(pdMS_TO_TICKS(10000)); // Раз в 10 секунд
  }
}
```

***Важные нюансы***

Накладные расходы: сбор статистики сам потребляет ресурсы. Не включайте его в продакшн-режиме без необходимости.

Синхронизация: при анализе загрузки обоих ядер учитывайте задержки между измерениями на разных ядрах.

Точность: процент загрузки, рассчитанный через vTaskGetRunTimeStats, может не учитывать время, потраченное на прерывания.

Конфигурация: убедитесь, что в sdkconfig включены нужные опции (особенно FREERTOS_GENERATE_RUN_TIME_STATS).

> ***CONFIG_FREERTOS_GENERATE_RUN_TIME_STATS — это настройка FreeRTOS, а не самой Arduino IDE. В стандартных библиотеках Arduino (особенно для ESP32) эта опция по умолчанию отключена. То есть если вы просто возьмёте и напишете в коде vTaskGetRunTimeStats(), компилятор выдаст ошибку «undefined reference» — функция просто не будет видна, потому что FreeRTOS не умеет собирать такую статистику. ***


### [Тест загрузки процессора ESP32 без специальной настройки](https://github.com/pglen/esp32_cpu_load/tree/master)

Мне нужен был ненавязчивый тест загрузки процессора, чтобы я мог просто вывести данные о загрузке процессора в терминал. И я не хотел включать подсистему статистики задач FreeRTOS.

Результат извлечен в этот проект.

Теория:

Мы создаем задачу с низким приоритетом (бездействие) и задачу с более высоким приоритетом (мониторинг).

Задача с низким приоритетом задерживается на 0 мс. Мы знаем, что задача передает управление процессором и не получает его обратно до следующего временного интервала. (10 мс)

 But ...
 
Если значение времени задержки равно нулю, планировщик пытается вернуть управление процессору, как только освободится временной интервал. Если управление не возвращается, это означает, что процессор занят другими «делами».

Суть метода измерения заключается в следующих трех строках:

```
    int64_t now = esp_timer_get_time();     // Time anchor
    vTaskDelay(0 / portTICK_RATE_MS);
    int64_t now2 = esp_timer_get_time();    // The diff is the time away.
```

Тестирование:

Каждые 10 секунд мы запускали интенсивный цикл. Вот что показывает консоль:

```

-0% -0% -0% -0% -0% -0% -0% work [[ 31% 100% ]] rest 38% -0% -0% -0% -0%
-0% -0% -0% -0% -0% work [[ 63% 100% ]] rest 6% -0% -0% -0% -0% -0% -0%
-0% -0% -0% work [[ 95% ]] rest 74% -0% -0% -0% -0% -0% -0% -0% -0%
-0% work [[ 27% 100% ]] rest 42% -0%
```

Обратите внимание, как красиво показано перекрытие скользящего временного окна.

### [Монитор использования процессора ESP32](https://github.com/Carbon225/esp32-perfmon/blob/master)

Дает приблизительное представление об использовании ядер Xtensa.

Работает путем установки хуков простоя и измерения частоты их вызова.

Использование ядра рассчитывается по формуле:
```
usage% = idle ticks since last measurement / expected idle ticks if core were idle * 100%
```

Ожидаемое количество тактов простоя было измерено при запуске пустой программы.

Ограничения:
Следует использовать только для пользовательской информации, а не в логике, требующей точных значений
Новые версии IDF могут оптимизировать производительность и, следовательно, привести к ошибкам в оценке использования.
Когда одно ядро загружено на 100 %, другое может показывать отрицательное значение
Использование:
Загрузить компонент в <project_dir>/components/
#include "esp32_perfmon.h"
Вызвать perfmon_start() один раз
